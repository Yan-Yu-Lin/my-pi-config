# Subagent Context Cultivation

Use this skill when you are considering starting or resuming a subagent, especially for complex work where the subagent may need to build understanding over time rather than answer a one-shot prompt.

A subagent is not merely a function call. It is another AI instance with its own context window, its own tool observations, and its own evolving understanding. It may have the same intelligence and tool access as the main agent; what it lacks is the main conversation context with the user. Treat it less like a junior worker and more like a peer coworker who has just entered the room. If you only hand it a command, it will optimize for the literal command. If you share the reasoning, the history, the uncertainty, and the direction of travel, you help it enter the right state to think with you.

The important idea is context cultivation. An AI model's next response is shaped by everything in its context: the words you give it, the files it reads, the tools it calls, its own previous messages, the assumptions it writes down, and the corrections it receives later. Multi-turn interaction with a subagent is not wasted overhead; it is how you deliberately build the subagent's working state. The goal is not only to transfer facts, but to shape the subagent's internal framing so that it understands why the work matters and how to reason about it.

Often the best first message to a subagent is not the final task. A better first message may be a warm-up: explain what the user is trying to accomplish, how the conversation got here, what you currently believe, what is uncertain, and what areas the subagent should inspect. Ask it to explore and report back its understanding before implementing. This lets you see what it notices, where it aligns with your reasoning, and where it is missing context. Once its understanding is good enough, resume the same subagent with the actual next step.

This is useful even when you, the main agent, already know the direction. You may know the goal, but the subagent does not yet have the context state needed to execute well. Sending it to inspect relevant files, docs, logs, or prior decisions helps it build its own map. Its report back is not just an answer; it is evidence of what state it has reached. If the report shows misunderstanding, steer it. If it found something you missed, update your own plan. If it is aligned, continue from there.

When prompting a subagent, do not only say what to do. Say why. Share the mental model you want it to inherit. Explain constraints, tradeoffs, prior attempts, user preferences, and what kind of uncertainty remains. You can ask the subagent to ask questions if it lacks context. You can ask it to avoid implementation until it has reported its understanding. You can resume it later with corrections and a more concrete task. The subagent session is resumable, so its accumulated context can become an asset.

A good subagent warm-up prompt sounds like a conversation with a collaborator: "Here is what we are trying to build. Here is how we got here. Here is what I think is true, but I want you to verify it. Please explore these areas, build your own understanding, and report back what you notice, what seems important, what is unclear, and what you would do next. Do not implement yet." A later resume prompt can then say: "Good, your understanding matches the direction. Now proceed with this specific implementation step..."

Use subagents for context-building branches, not only for task completion. A subagent can scout, reason, compare approaches, audit assumptions, or hold a separate thread of investigation. The main agent coordinates these peer contexts. The deeper skill is knowing when to create another context window and how to cultivate it until it becomes useful.
